# Картотека на клиенти

Уеб приложение за търсене, добавяне и изтриване на клиенти, с 15 персонализируеми
параметъра на клиент, за които се пази пълна история от стойности (всяка нова
стойност се записва с дата и час, без да трие старите).

Технологии: React + Vite (front-end, hostнат безплатно в Netlify/Vercel) +
Supabase (безплатна Postgres база данни + Storage за снимки на клиенти).

---

## 1. Създай безплатна база данни (Supabase)

1. Отиди на **https://supabase.com** → Sign up → New project (безплатен план).
2. Изчакай проектът да се създаде (около 1-2 минути).
3. В лявото меню отвори **SQL Editor** → New query.
4. Отвори файла `supabase/schema.sql` от този проект, копирай цялото
   съдържание и го изпълни (бутон **Run**). Това създава трите таблици
   (`clients`, `parameters`, `parameter_entries`), 15-те начални
   параметъра и storage bucket-а за снимки.
5. В лявото меню → **Project Settings → API**. Ще ти трябват:
   - **Project URL** (пример: `https://abcdxyz.supabase.co`)
   - **anon public** ключ

---

## 2. Настрой локалните променливи

1. Копирай `.env.example` в нов файл `.env`.
2. Попълни двете стойности от стъпка 1.5:

```
VITE_SUPABASE_URL=https://твоя-проект.supabase.co
VITE_SUPABASE_ANON_KEY=твоят-anon-ключ
```

---

## 3. Пусни го локално (по желание, за проверка)

```bash
npm install
npm run dev
```

Отвори връзката, която показва терминалът (обикновено `http://localhost:5173`).

---

## 4. Качи проекта в GitHub

1. Създай нов празен repository в GitHub.
2. В папката на проекта:

```bash
git init
git add .
git commit -m "Първоначален вариант на картотеката"
git branch -M main
git remote add origin https://github.com/ТВОЯ-ПОТРЕБИТЕЛ/ИМЕ-НА-REPO.git
git push -u origin main
```

(`.env` е в `.gitignore` и няма да се качи — правилно, ключовете се задават
отделно в хостинга, виж стъпка 5.)

---

## 5. Деплой безплатно (избери Netlify **или** Vercel)

### Вариант А — Netlify
1. **https://app.netlify.com** → Add new site → Import an existing project → избери GitHub repo-то.
2. Build command: `npm run build`, Publish directory: `dist` (вече е зададено в `netlify.toml`).
3. Site settings → Environment variables → добави `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`.
4. Deploy site. След 1-2 минути ще получиш безплатен адрес от типа `https://random-name.netlify.app`.

### Вариант Б — Vercel
1. **https://vercel.com** → Add New → Project → избери GitHub repo-то (Vite се разпознава автоматично).
2. Settings → Environment Variables → добави `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`.
3. Deploy. Ще получиш адрес от типа `https://ИМЕ.vercel.app`.

## Обновяване на вече съществуваща база (миграция)

Ако вече си пуснал сайта преди и имаш добавени клиенти, **не пускай отново
целия `schema.sql`** — вместо това отвори Supabase → SQL Editor → New query,
копирай съдържанието на `supabase/migration_2.sql` и го изпълни. Той:

- разделя 15-те параметъра на 2 групи (10 „Танита“ + 5 „Мерки на тялото“),
- сменя записа на стойностите да пази само дата, без час,
- ако параметрите все още имат старите имена „Параметър 1“...„Параметър 15“,
  ги заменя със смислени имена по подразбиране (само тези, които не си
  преименувал вече от Настройки).

Нищо от вече въведените клиенти или история не се трие.

Ако вече си пуснал и предишната версия (с групите Tanita/Мерки на тялото),
пусни допълнително и `supabase/migration_3.sql` — той просто маркира всички
параметри като числови (типът вече не се избира от Настройки, всичко е число).

## Вход с потребители и роли (важно - изисква еднократна настройка)

Сайтът вече изисква вход и има 3 вида акаунти:

- **Администратор** - пълен достъп до всичко.
- **Треньор** - може да създава нови акаунти (треньори и клиенти) "под себе си"
  и вижда/управлява само клиентите и под-акаунтите, които той (или треньор,
  когото той е създал) е създал.
- **Клиент** - вижда собствените си "Танита измервания" само за преглед (без
  редакция), а "Мерки на тялото" може и да преглежда, и да добавя/редактира/
  трие.

Създаването на нов акаунт изисква специални (администраторски) права в базата,
затова **не може да стане само от браузъра** - иначе всеки посетител би могъл
да открадне тези права. Затова има малка сървърна функция (Supabase Edge
Function), която го върши безопасно. Трябва да я качиш веднъж:

### А) Пусни миграциите в базата

Supabase → SQL Editor → New query → изпълни **последователно**:
1. `supabase/migration_4.sql`
2. `supabase/migration_5.sql` (добавя вход с потребителско име вместо имейл)

### Б) Качи сървърната функция (еднократно, или пак ако вече си я качвал по-рано)

Нужен ти е [Supabase CLI](https://supabase.com/docs/guides/cli):

```bash
npm install -g supabase
supabase login
supabase link --project-ref ТВОЯТ-PROJECT-REF   # виждаш го в URL адреса на проекта
supabase functions deploy manage-account
```

Ако вече си я качвал преди (за версията с имейл), просто пусни отново
`supabase functions deploy manage-account` — ще презапише старата версия с
новата, която ползва потребителско име и позволява смяна на пароли.

Това е всичко — **не** е нужно да задаваш `SUPABASE_SERVICE_ROLE_KEY` ръчно.
Supabase го подава автоматично на всяка Edge Function (заедно с `SUPABASE_URL`
и `SUPABASE_ANON_KEY`), затова CLI дори отказва да презапише имена, започващи
с `SUPABASE_` — ако видиш съобщение от рода на "Env name cannot start with
SUPABASE_, skipping", това е очаквано и функцията вече работи нормално.

### В) Създай първия администраторски акаунт (еднократно, ръчно)

Новите акаунти вече се създават с потребителско име (не реален имейл), но
самият Supabase Auth изисква технически поле "имейл" отдолу — затова за
първия администратор просто въвеждаш нещо от вида `потребителскоиме@clientdb.local`:

1. Supabase Dashboard → Authentication → Users → Add user:
   - Email: `твоетоиме@clientdb.local` (измисли си потребителско име, залепи `@clientdb.local`)
   - Password: избери парола
   - Auto Confirm User: ON
2. Копирай новия User UID.
3. SQL Editor → изпълни (замени стойностите):
   ```sql
   insert into profiles (id, role, full_name, username, email)
   values ('ПОСТАВИ-USER-UID-ТУК', 'admin', 'Твоето име', 'твоетоиме', 'твоетоиме@clientdb.local');
   ```
4. Влез в сайта с потребителско име `твоетоиме` и паролата → вече си
   администратор → от менюто "Акаунти" можеш да създаваш треньори и клиенти
   (вече само с потребителско име, без имейл) и да сменяш паролите им по
   всяко време с бутона 🔑.

**Забележка за паролите:** никой (дори администраторът) не може да "види"
вече зададена парола — те се пазят криптирани и необратими, това важи за
всяка сериозна система. Бутонът 🔑 позволява на администратора (и на
треньор, за акаунтите, които той е създал) да **зададе нова** парола на
потребител по всяко време, което на практика решава същия проблем (връщане
на достъп при забравена парола).

## Език на сайта

Горе вдясно на всяка страница има два флага — 🇧🇬 за български и 🇬🇧 за английски.
Целият интерфейс (менюта, бутони, форми) се превежда веднага при избор.
Избраният език се пази в браузъра на всеки посетител поотделно. Имената на
клиентите и на параметрите, които въвеждаш сам, не се превеждат автоматично.

## Структура на данните

- **clients** — лична информация (име, адрес, телефон, имейл, снимка, дата на
  раждане, пол, бележки). Възрастта се изчислява автоматично от датата на раждане.
- **parameters** — 15-те проследявани параметъра, разделени на 2 групи:
  `tanita` (10, везна Tanita) и `body` (5, обиколки на тялото), всяка на своя
  страница от профила на клиента. Имената и типът (число/текст) се редактират
  от бутона **„Настройки на параметрите"** в приложението.
- **parameter_entries** — всяка добавена стойност е отделен ред с `recorded_at`
  (само дата, без час — може да се зададе ръчно при добавяне). Старите стойности
  никога не се трият/презаписват — това е пълната история на клиента.

## Сигурност

Достъпът вече е ограничен през вход + роли (виж секцията по-горе), със
съответните права, прилагани на ниво база данни (RLS) - не само във
фронтенда. `SUPABASE_SERVICE_ROLE_KEY` съществува само в сървърната функция
`manage-account`, никога във фронтенда/GitHub.
